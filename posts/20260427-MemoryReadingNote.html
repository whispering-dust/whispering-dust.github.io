大模型Memory

「纯干货」几万字都讲不明白的Memory架构与思考

提示：这篇文章更多是阅读过程中的一些随想与摘录，结构上可能不够严谨；如果希望获取更系统的观点，推荐直接阅读原文。

关注Memory有一段时间了，现在对于Memory的研究五花八门（logit 调制、工具化与 RL 训练、记忆关注 Memory 已经有一段时间了。最近这方面的研究可以说是百花齐放：从 logit 调制、工具化结合 RL 训练，到记忆固化、技能化，再到协议层的对象模型与闭环设计，各种“记忆架构”层出不穷，也让整个系统变得越来越复杂。

与此同时，不少工作也在探索如何通过训练，让模型自主决定“何时存、何时取”。不过从结果来看，这类方法目前仍然更像是实验室中的探索性方案。工业界的实践反而相对克制，常见的工业级方案仍然以 Mem0 或围绕 Hardness Engineering 的设计为主。

最近读到一篇系统总结 Memory 的文章，颇有启发，这里整理一下自己的几点理解。

作者提出了一个核心判断：从长期来看，参数化 Memory 的潜在上限更高；但在当前工程条件与资源约束下，非参数化 Memory 更容易落地。 与此同时，作者也在持续探索一个关键问题——如何设计非参数化 Memory，使其尽可能逼近参数化方案的效果上限。

文章在后面详细展开“为什么参数化 Memory 的上限更高”。在我看来，其核心原因在于：参数化 Memory 本质上是将“记忆”转化为“模型能力本身”，从而摆脱了检索带宽与上下文窗口的限制，并具备更强的跨层融合与泛化能力。

相比之下，非参数化 Memory（如 RAG、向量库、本地记忆等）的基本范式是：先检索 → 再拼接到上下文 → 最后由模型进行理解。这种流程中存在一个难以消除的鸿沟：模型始终是在“读取记忆”，而不是“直接用记忆进行思考”。

而参数化 Memory 则直接体现在模型权重之中，并参与每一层的计算过程。这带来了几个关键差异：

  • 信息可以在多层网络中被反复加工与重组
  • 能够与已有知识进行非线性融合
  • 不受上下文窗口限制

但其代价同样明显：参数一旦固化，更新成本极高；频繁微调容易引发“灾难性遗忘”；同时整体可解释性也相对较弱。

回到工程实践，当前主流 Memory 系统更强调“可插拔、可更新、可持续学习”等属性。在这个语境下，参数化 Memory 更接近人类大脑中的“潜意识”或“固有认知”——即沉淀在模型参数或潜在空间中的能力本身。

当然，也存在一些外接的“参数化 Memory”设计，但这类方案通常与特定模型深度耦合，一旦模型替换，效果往往难以迁移。从这个角度看，让非参数化 Memory 在工程上不断逼近参数化方案的表现，确实是一条更现实的路径。

进一步延伸，一个值得思考的问题是：是否可以构建一种类似 MoE 的 Memory 机制，将记忆模块以内嵌的方式融入模型，在需要时动态激活？以及，这样的机制是否能够在持续更新的同时，不破坏已有能力的稳定性？这或许指向了 Memory 设计中一个更具潜力的方向。

1 Memory定义 ​

回到文章本身，作者提出了三个核心命题，我个人非常认同：

  • Memory 不是“存储”，而是可被决策利用的外部状态（external state）

    文中强调，Memory 不应只是对历史的被动记录，更关键的是：这些历史必须能够实际影响当前决策。换句话说，Memory 的价值不在“存了什么”，而在“能否被用来改变行为”。

  • Memory 的最小闭包不是“文档块”，而是 (Ledger, Views, Policy) 三件套

    作者将 Memory 抽象为一个“可审计的状态机”，这个总结我觉得非常精准。具体可以拆为三层：

    • Raw Ledger：以追加为主的底层记录，完整保存每一次写入、更新、删除等操作，关注“发生了什么”
    • Derived Views：面向检索与推理的派生表示，例如向量索引、keyword/hybrid 检索、知识图谱（KG / 时序知识图谱）、timeline、skill index 等，本质上决定了“如何组织与访问记忆”
    • Policy：用于控制读写与更新策略，包括何时读取、读取多少、何时写入、如何更新以及如何遗忘；并且这些决策需要被显式化为可记录、可回放的 Action 序列（如 ADD / UPDATE / DELETE / NONE 等）。这一层也是当前研究最为集中的部分

  • Memory 的基本单位应当是 event 序列，但“直接使用 event 流”并不足以构成系统

    我比较认同作者的一个判断：event 序列是“真相来源”，但过于底层。如果仅仅存储 event，你得到的是一套可审计的历史记录，而不是一个可用的记忆系统。

    只有在此基础上引入：

    • View（对 event 的组织与抽象）
    • Policy（对读写与更新的控制）

    才能真正形成具备实用价值的 Memory 机制。

在这些命题之上，我感觉作者其实隐含表达了一个非常关键的原则：Memory 必须是“可溯源（provenance）、可回滚、可观测”的系统。

这背后其实对应的是对“记忆可靠性”的要求——只有当 Memory 的来源清晰、变化可追踪、行为可解释时，我们才有可能在系统层面真正信任它。

某种程度上，这也解释了为什么当前很多 Memory 实现，倾向于以 Markdown 等人类可读的形式进行存储：这并不只是工程上的权宜之计，而是一种在“可审计性”与“可理解性”之间取得平衡的选择。

2 System 1 + System 2 的设计 ​

这一部分的讨论，其实很好地呼应了我在开头提到的一个判断：相比仅依赖单一的 System 1，引入一个外部、可插拔的 Memory 体系具有明显优势。 同时，文章提出了一个重要观点——记忆能力与 LLM 本身的其他 Agent 能力在很大程度上是“相对正交”的。

从经验现象看：同一个 base model 在不做记忆特化训练时，接入不同的 RAG / 长期记忆策略就能显著改变长程任务表现；而同一套记忆 infra（raw_ledger + views + 检索策略）也往往能服务多种不同的 base model（可迁移、可插拔）。

不过，这种“正交性”并不是绝对的。文章中也特别强调了一个关键点：

检索噪声、错误召回以及时序冲突，会直接干扰推理过程（导致推理路径偏移甚至引发幻觉）；同时，“何时检索、检索多少、何时写入与更新”等策略，本身也依赖于 Agent 的自我评估与规划能力。因此，更准确的描述应当是：**“相对正交，但存在可控的交叉项”。**而 System 2 的 observability、provenance 以及 sandbox / A-B 机制，其意义正是在于将这些交叉项显式化，使其可诊断、可迭代。

从这个角度看，Memory 的一个核心问题，其实变成了：如何优化 System 1 与 System 2 之间的交互质量，尤其是检索与读写策略，把在线适应的主战场从模型权重挪到外部记忆状态与控制策略上，并让它可观测、可回放、可 A/B。

在实际研究中，这一方向已经相当“内卷”。一个比较有代表性的工作是 Memory-R1，它尝试通过 RL 训练模型，使其学会在合适的时机进行检索，以及对记忆执行增删改查等操作。

在早期，传统 RAG 更多被用作一个“事实校验层”，用于对抗幻觉或提供外部知识增强；但在当前的 Agentic Memory 语境下，研究者越来越倾向于让模型自主决定记忆的生成与使用，甚至在较少人工干预的情况下完成复杂任务。

这也带来了新的风险：一旦错误或有害信息被写入 Memory，并在后续流程中被反复引用，就可能沿着链式过程不断放大，最终导致任务偏离甚至整个工作流崩溃。

因此在我看来，“何时存”背后其实还隐藏着一个更关键的问题：如何避免“记忆中毒”（memory poisoning）。换句话说，系统不仅需要决定“是否写入”，还需要具备对记忆内容进行质疑与筛选的能力，而不是默认信任所有生成结果。

3 参数化Memory和非参数化Memory ​

在这一部分，文章详细讨论了参数化的问题，首先要明确定义：

  • 参数化记忆（parametric memory）：经验被写进模型权重。训练 / 微调就是把历史数据编译进权重的过程，推理时直接用更新后的模型，不需要额外检索。
  • 非参数化记忆（non-parametric memory）：经验被写在外部状态里（ledger + views + skill pool + 索引结构）。写入由 policy 决定写哪些、怎么写；推理时通过检索、汇聚、注入等环节让外部状态影响输出。

文章里，把外置记忆抽象为**“对模型输出施加一个可控的修正项 Δ”**，模型的决策写成 logits（或 action distribution），Δ相当于一个可控偏执可以改变决策的分布，然后做了很多的理论上的讨论和证明。

对于“外挂记忆”的上限，文章认为由下面三类瓶颈共同决定：

  • 接口带宽（Memory → System 1 的注入容量）
  • 检索与聚合误差（views 的近似误差）
  • policy 的可学习性与可控性（什么时候写/读、写什么、信谁）

4 Memory System 控制层/策略层（policy） ​

接下来文章开始重点探讨policy部分，这也是近期文章大量涌现的方向，也就是Agentic Memory。

这里先强调一下定义：记忆操作彻底工具化，整合进 Agent 的动作空间。让模型像控制手臂一样控制记忆，而非被动接收上下文。

  • 综合推理阶段：结合 LTM 检索和 STM 管理，解决复杂的长程任务（如 ALFWorld, HotpotQA）。

除了上述工具化的训练方式，即让大模型把记忆当做工具，并学会去调用这个工具之外。文章还提到了一个优化System2的Policy的工作：InfMem。

这篇工作为了解决长文档推理中的"迷失中间"（Lost-in-the-Middle）问题，引入了 PreThink-Retrieve-Write 协议，显式模拟了慢思考过程

  • PreThink（预思考）：在检索前，Agent 先评估当前内部权重知识是否足以回答问题。这有效减少了不必要的外部检索，降低了延迟 。
  • Adaptive Early Stopping（自适应早停）：传统 RAG 往往检索固定数量的块（Top-K），而 InfMem 训练了一个策略网络，当累积证据置信度达到阈值时立即停止检索。实验显示，这一机制将推理速度提升了 3.9 倍 ，极大地逼近了参数化模型的即时响应速度。
  • 训练范式（SFT-to-RL）：InfMem 通过从监督微调（SFT）过渡到强化学习（RL），直接优化检索和记忆更新决策，确保每一次外部交互都是为了最大化最终答案的准确率。

4.2 Memory 的时序问题 ​

文章点明了一个重要的问题：

参数化记忆的一个重大缺陷是"静态世界观"——一旦训练完成，权重就趋于固化，难以自然地表达随时间变化的事实”。

因此需要引入时序的概念。而在引入“时序”这个概念之后，Memory OS 的三件套 **(Raw Ledger, Views, Policy)**天然就会发生变化，更为复杂：

  • Raw Ledger：不止要记录“系统何时什么写入”（transaction time），还要能表达“何时写入”，以及**"该事实在世界中何时为真"（valid time）**
  • Views：检索时要考虑“某个时间维度下语境是否成立”
  • Policy：要求 policy 在需要历史时能显式触发 historical/all，并让 EvidencePack 承载"冲突并存"的证据结构

此外时序的问题不仅仅依赖模型，还需要提升到架构层面，因为：

  1. LLM 对时间天然不敏感（尤其是隐含时间、相对时间、跨时区）
  2. 纯语义检索会把"过去的高相似事实"错当成"现在仍为真"
  3. 决策层出现"过时事实复活"、以及"被纠正的事实仍反复召回"
  4. 解决方案不是更强 prompt，而是：以 bi-temporal + time-sliced recall 把"何时为真"变成检索与聚合的硬约束，并让 policy 可控、可回放。

需要补充的是，**bi-temporal（双时间）**是数据库里的概念，指一条数据有两个时间：

  • 事件时间（event time）：事情“实际发生”的时间
  • 记录时间（system time）：系统“知道这件事”的时间

time-sliced recall指按时间切片的检索。

文章的这整段其实在强调一个被低估的问题：Memory 的核心不是“存什么”，而是“什么时候成立”。

现在很多系统用语义相似度做检索，但语义不会区分时间，导致“过去的正确”不断污染“现在的判断”。这篇的核心主张是把时间提升为系统的硬约束——所有检索必须先满足时间，再谈相关性。本质上，是把 Memory 从“知识集合”变成“带时间坐标的事实系统”。我觉得这个视角挺关键的，因为它解释了为什么很多 AI 看起来不是不会推理，而是“时间线错了”。

4.3 Memory的结构问题 ​

在解决完**“记什么 & 什么时候为真”**的问题之后，下一个问题就是，“记忆应该长成什么样”？

文章把 Memory 从“知识库”进一步推进到“能力系统”。

  1. 首先讨论为什么要分层：核心是因为 **不同类型的信息，不应该用同一种方式存。**文本、图谱、经验分别承担证据、关系和抽象能力。

    • Graph Memory（图） → 存关系（谁和谁有关）
    • Experience Memory（经验） → 存模式（“通常这样会成功”）
    • Passage Memory（文本） → 存原始证据（原话是什么）

  2. 接下来是讨论，什么样的记忆是有价值的：关键的转变是从 declarative memory（知道什么）走向 procedural memory（知道怎么做）。像 ProcMEM 这种工作，本质是在把成功的推理路径固化为可执行技能，而不是每次临时生成。而这也就是近期频繁出现的Skill

  3. 最后落到工程问题：这些记忆怎么进入模型。传统做法是转成文本再喂，但这有损耗，所以开始出现 memory tokens 这种“机器原生记忆”的方向。

总而言之，Memory 系统的演进不是“多存点信息”，而是三步升级：

  1. 结构升级：记忆要分层（文本 / 关系 / 模式）
  2. 能力升级：从“存事实”变成“存技能”
  3. 接口升级：让记忆以更低损耗方式进入模型（甚至非文本）

同时必须保证：可验证、可治理、可溯源。

5 总结 ​

回看整篇文章，它其实不是在介绍某一种具体的 Memory 实现方案，而是在尝试回答一个更底层的问题：

一个可持续演化的 Memory System，本质上应该长成什么样？

作者最终给出的答案，并不是某个具体算法，而是一套类似操作系统的抽象结构——Memory 不再是一个“组件”，而是一个完整的系统架构。再重新梳理一下文章的核心内容：

【1】 Memory 的本质：从“存储”到“闭环系统”

文章最核心的一个认知转变是：

Memory 不是存储，而是一个可参与决策、可持续演化的闭环系统。

这个闭环可以抽象为：

Raw Ledger → Views → Policy → Commit → Provenance

  • Ledger 保证“发生过什么”的权威记录（真实来源）
  • Views 决定“如何被使用”（检索、抽象、组织）
  • Policy 决定“何时读写、信谁、如何更新”（控制层）
  • Commit / Provenance 保证系统可回放、可审计、可迭代

【2】System 2 的意义：把“学习与适应”外置

文章一个很重要的工程判断是：

记忆能力与模型能力是“相对正交”的。

这意味着：

  • Memory 可以作为 System 2 外置系统存在
  • 不依赖模型权重更新，也能持续优化行为

其核心价值在于：

  • 可插拔（不同模型复用同一套 Memory）
  • 可观测（行为可分析）
  • 可回放（决策可复现）
  • 可 A/B（策略可优化）

本质上是把“学习”从权重转移到外部状态 + policy上。

【3】非参数化 Memory 的本质瓶颈

文章给了一个很清晰的上限分析：

非参数化 Memory 的能力，受三类瓶颈限制：

  • 接口带宽（能喂多少信息给模型）
  • 检索与聚合误差（view 是否接近真实）
  • policy 可学习性（何时读写、如何决策）

其中一个非常关键的洞察是：

policy 往往是最被低估、但最核心的瓶颈。

【4】时序：从“附加信息”升级为“结构维度”

这是全文最关键的升级之一：

Memory 的核心问题不是“存什么”，而是“什么时候为真”。

如果没有时序约束：

  • 过去的事实会污染当前决策
  • 已纠正信息会反复出现
  • 系统会逐渐“时间错乱”

因此必须引入：

  • bi-temporal（双时间）
  • time-sliced recall（时间切片检索）

并把“何时为真”变成系统级硬约束，而不是交给模型猜。

【5】Memory 形态的演进：从知识到能力

在解决“存什么 & 何时为真”之后，文章进一步推进：

Memory 不只是知识库，而是能力系统。

演进路径可以总结为三步：

  1. 结构升级：分层记忆（文本 / 图谱 / 模式）
  2. 能力升级：从 declarative（是什么）→ procedural（怎么做）
  3. 接口升级：从文本注入 → memory tokens（机器原生）

其中最关键的转变是：

把“成功经验”固化为可执行技能（skill），而不是每次重新推理。

同时要求：

  • 可执行
  • 可验证
  • 可治理

【6】最终抽象：Memory OS 的五层架构

文章最后给出的，是一个非常“操作系统化”的抽象：

  • Kernel（控制层）：负责调度与策略（何时读写）
  • File System（存储层）：Ledger + Views（数据与表示）
  • Executable（技能层）：可复用的程序性记忆
  • Interface（接口层）：Memory 与模型的桥接
  • Learning Engine（学习层）：持续优化策略与状态

这五层不是实现方案，而是：

“一个成熟 Memory 系统必须具备的最小模块集合”
